# SNVLA 課題調査と改善設計

- 日付: 2026-07-11
- 対象: `lerobot-policy-snvla`（SNVLA論文の再現実装、LeRobot 0.6ベース）
- 成果物の位置づけ: 課題の全体調査と改善案の設計ドキュメント（実装は含まない）
- 評価基準: 研究としての新規性 / タスク性能・信頼性 / 実用性（遅延・スケーラビリティ）
- 前提条件: 制約なし（モデルアーキテクチャ・タスク・環境とも変更可）。設計の帰結として、P0によりベースモデルはMolmoAct2系を前提とする

## 1. 背景 — 現行SNVLAの要約

SNVLA (Self-Narrating VLA) は π0.5（PaliGemma 2B + action expert 300M）をベースに、
ロボット自身が行動を自然言語で実況し、その履歴を短期記憶としてプロンプトに再注入する
ことで Perceptual Aliasing を解決するVLAである。豆すくいカウントタスクで成功率96.7%
（π0.5ベースライン33.3%、実況無効化ablationは0.0%）を達成した。

### 1.1 推論フロー（現行実装）

チャンク境界ごとに以下を完全シーケンシャルに実行する
（`src/lerobot_policy_snvla/modeling_snvla.py` の `select_action`）:

1. プロンプト構築: `Task: ... / State: <離散化状態> / Progress: <実況履歴連結>`
2. prefill（画像+テキストのフル前向き計算、KVキャッシュ取得）
3. モード決定: 最終トークンのlogitsを BON (id 50) / BOA (id 51) の2トークンに制限してargmax
4. BONなら実況を自己回帰デコード（最大50トークン）、履歴 `_previous_narrations` に追加
5. BOAトークンを前向き計算し、flow matchingで行動チャンク（chunk_size=50、実行は
   n_action_steps=15 ≒ 0.5s @30fps）を生成

### 1.2 学習構成（現行実装）

`processor_snvla.py` の `SNVLAPrepareTrainingTokenizerProcessorStep`:

- コンテキスト: `Task / State / Progress`（`previous_narrations` は教師データの真値を使用）
- ターゲット: 実況フレームは `[BON] 実況 [EOS] [BOA]`（テキストCE損失、重み5.0）、
  非実況フレームは `[BOA]` のみ。全サンプルで行動の拡散損失を併用
- BON/BOAはPaliGemma語彙の未使用トークン (`<unused>` 枠) を転用

### 1.3 実測値（論文draft `inference_time_analysis.md`）

| 種別 | 平均推論時間 |
|---|---|
| 行動のみのフレーム | 0.415 s |
| 実況生成フレーム | 0.659 s（+0.244 s） |

## 3. 関連研究ポジショニング

新規性評価のため、2025–2026年の近接研究を調査した。

### 3.1 モード切替型の統合VLA

- **OneTwoVLA** (arXiv:2505.11917): SNVLAに最も近い。単一モデルが [BOR]/[BOA] トークンで
  推論/行動を適応切替し、決定トークンに二値分類損失を追加。推論テキストにはシーン記述・
  計画・現在ステップが含まれ、実質的に進捗記憶としても働く。VLデータの合成パイプラインで
  co-trainingする。**「特殊トークンによるモード切替」はもはや確立された手法であり、
  SNVLA固有の新規性ではない。**
- **StreamVLA** (arXiv:2602.01100): サブタスク遷移時のみSystem-2推論を行い、遷移間は
  事前計算済みintentをflow matchingヘッドに固定して自己回帰デコードを回避。
  72%のタイムステップでデコードを省き、遅延48%削減。
- **ECoT / CoT-VLA / ThinkAct 等**: 行動前に明示的な思考テキスト/視覚サブゴールを生成する
  think-before-act系。**Latent Reasoning VLA** (arXiv:2602.01166) は思考を潜在表現に
  内在化して遅延を最大90%削減。
- **MolmoAct2** (arXiv:2605.02881): embodied-reasoning VLM（Molmo2-ER）を出発点に、
  flow-matching連続Action Expert・離散action・depth reasoningの複数生成経路を持つVLA。
  公開checkpointが自然言語生成能力を保持していることはローカル実推論で検証済み。
  本設計ではP0の移行先として採用する（詳細はP0）。

### 3.2 推論遅延対策

- **Real-Time Chunking (RTC)** (arXiv:2506.07339): 現チャンク実行中に次チャンクを生成し、
  実行確定分をfreeze、残りをinpaintする推論時アルゴリズム。再学習不要で任意の
  flow/diffusion系VLAに適用可能。本実装のconfigには `rtc_config` が既に存在する。

### 3.3 自動アノテーション

- VLMによるデモのサブタスク分割+密ラベリングは活発（DeMiAn、RoboFine-VLM等）。
  ただし汎用ベンチマークではend-to-endのサブタスク分割はF1 0.168程度と低く
  （Macrodata Labsのベンチマーク）、VLM単独のタイミング検出は信頼できない。

### 3.4 SNVLAの新規性の残余と、次に主張すべきこと

SNVLAの差別化点は「**生成した実況テキストを明示的な短期記憶としてプロンプトに再注入し、
Perceptual Aliasing（カウントタスク）を実機で解いた**」部分に絞られる。次の研究で
新規性を主張できる方向は:

1. **自己教師による実況アノテーション拡張**（提案P1）: モデル自身を実況タイミング
   予測器として育て、実況なしデータへの自動アノテータにする2段階スキームは
   調査範囲で存在しない
2. **行動を止めない実況**（提案P2）: OneTwoVLA（推論中は行動停止）とも
   StreamVLA（遷移時のみ推論）とも異なる。生成途中の実況KVを行動生成が参照する
   機構まで含めれば差別化はさらに明確になる

## 4. 重点提案

### P0: 設計前提 — ベースモデルの移行（π0.5 → MolmoAct2）

対応課題: P1の前提（言語生成能力）、C4（接地の質）

#### 動機

現行ベースの `lerobot/pi05_base` はロボットデータへのfine-tuneで自由言語の生成能力を
失っており、自己実況という手法の前提と根本的に相性が悪い。回復策（VL co-training）を
講じるより、**言語生成能力が実証済みのVLAから出発する**方が抜本的である。

AllenAIのMolmoAct2系は、公開8チェックポイントすべてでLM headの実在と自然言語生成が
実推論で検証されている（検証記録: `$HOME/Workspaces/molmoact2-pg/README.md`）。
また、出発点のMolmo2-ERはembodied reasoningデータで学習されており、視覚接地・空間推論の
強さが実況の観測整合性（C4: ハルシネーション低減）にも寄与すると期待できる。

#### 移行先の選定

- **主案: `allenai/MolmoAct2-SO100_101`**。理由は2つ:
  (i) embodimentが論文実験のSO-101と一致しロボット事前学習を直接活かせる、
  (ii) robot-specificチェックポイントの中で、ロボット画像入力に対しても
  自然言語生成を維持している（DROID/BimanualYAMはロボット画像でaction token列に
  遷移するが、SO100_101は自然言語を保った）。
- 比較候補: `allenai/MolmoAct2`（基盤、embodiment非依存）、
  `allenai/MolmoAct2-Think`（depth reasoning付き。ただし「Think」は自然言語CoTでは
  なく離散depth codeによる幾何条件付けであることに注意）。

#### アーキテクチャ適合性

- MolmoAct2は**flow-matching連続Action ExpertがVLMの各層KVキャッシュを条件に行動を
  生成する**構造であり、π0.5のaction expertと同型。SNVLAの設計
  （`Progress:` 履歴つきprefix → モード決定 → 実況デコード → KV条件付け行動生成）は
  そのまま移植できる。
- Think系は「推論トークン（depth）のKVでAction Expertを条件付ける」経路を既に持ち、
  実況トークンのKV条件付けと構造的に相似。P2の「生成途中の実況KV参照」とも整合する。
- 実況デコード時は、語彙を共有する離散action token（`<action_i>`）をlogitマスクで
  禁止し、テキストへの混入（robot-specific checkpointで観測された現象）を防ぐ。

#### 移行コストとリスク

- **LeRobot統合**: LeRobot形式チェックポイント（`MolmoAct2-LIBERO-LeRobot`）が存在し
  policy実装の前例がある。本プラグインは `PI05Policy` 継承から MolmoAct2系policyへの
  移植が必要。
- **依存関係**: MolmoAct2のconfigはTransformers 5.x系を要求する（4.57系では
  processorのロードに失敗する実績あり）。LeRobot 0.6との共存可否を移行時に確認する。
- **資源**: bf16推論で約11 GiB（RTX 3090で検証済み）。学習時のFSDP構成は再見積もりが
  必要。
- **言語能力の再喪失**: 我々のロボットデータでのfine-tuneにより、DROID/YAMで観測された
  ようなテキスト/action tokenの競合・言語能力の侵食が起こり得る。対策として
  (i) VLデータのco-training混合（P1参照）、(ii) 公式training codeが対応する
  LM head等の選択的freeze、(iii) P1-E2（言語能力の定量監視）を移行後も常時適用する。

### P1: 実況タイミング予測を核とする自己教師アノテーション拡張

対応課題: C3（主）、C1・C6（副次的に解消）

#### 動機と全体像

SNVLAを「実況タイミング予測モデル」として育てることで、モデル自身を実況なしデータへの
自動アノテータにする。学習は2段階になる:

- **第1段階（シード教師あり）**: 少量（10〜25%程度を想定、P1-E1で掃引）の実況テキスト
  +タイミング付きシードデータで、実況生成と停止判定を教師あり学習する
- **第2段階（自己教師拡張）**: 実況なしの大量デモに対し、第1段階モデルが推論時と
  同一の手順で実況を生成して擬似ラベル化し、検証機構（後述）を通過したものを教師に
  加えて再学習する。ラウンドは反復可能

これにより、追加データの収集はキー押下すら不要（実況なしの通常テレオペのみ）になり、
テキスト教師強制によるexposure bias（C6）も第2段階で構造的に緩和される
（学習時の履歴分布が推論時の自己生成分布に一致していくため）。

#### モード決定機構 — EOS周辺確率方式（BON/BOAの廃止）

BON/BOA特殊トークンを廃止し、決定を語彙分布そのものに統合する:

- prefill後の次トークン分布で **P([EOS]) vs それ以外の確率和** を比較する。
  EOSなら「今は語ることがない」= 行動生成へ、非EOSならそのまま実況を
  自己回帰デコードし、EOSで停止して行動生成へ移る。
  「EOSの後は行動生成」という規約により、BOAトークンは不要になる
  （EOSトークンのKVが行動生成のトリガー位置を兼ねる）。
- タイミングのみの教師データ（第2段階で内容正確性のみ不合格だったサンプル等）に
  対しては、テキスト内容を指定しない二値損失 `-log P(EOS)`（行動フレーム）/
  `-log(1 − P(EOS))`（実況フレーム）でタイミングだけを教師できる。
- クラス不均衡（実況フレームは全体の数%）は正例重み付けまたはfocal lossで対処。

代替案として検討した **二値タイミングヘッド**（prefix最終隠れ状態からのlinear分類器）
との比較:

| 観点 | EOS周辺確率方式（主案） | 二値タイミングヘッド |
|---|---|---|
| 追加パラメータ | なし | linear 1層 |
| 決定と生成の一貫性 | 一体（最初の生成トークンが決定そのもの） | 分離（ヘッドがnarrate判定してもLMが即EOSを出す不整合があり得る） |
| タイミングのみ教師 | EOS周辺確率への二値損失で可能 | BCEで可能 |
| VLM凍結との両立 | 不可（LM本体に勾配が必要、LoRA等の部分解凍で緩和） | 凍結のまま学習可能 |

決定と生成のあいだに縫い目がないことを重視し、EOS方式を主案とする。
VLM本体の完全凍結が必要になった場合のみ、タイミングヘッドをフォールバックとして採用する。

#### 言語生成能力の確保 — P0を前提とした維持策

言語生成能力の確保はP0（MolmoAct2への移行）で根本的に解決する。本提案側の課題は
「我々のデータでのfine-tune中に能力を**維持**すること」に変わり、対応は:

- **(a) VLデータのco-training混合（主対策）**: 実況データ・画像キャプション等の
  vision-languageデータをロボットデータと混合してfine-tuneし、言語能力の侵食
  （MolmoAct2のDROID/YAM checkpointで観測されたaction tokenとの競合）を防ぐ。
  シードデータ自体もVLデータとして働く。混合比が設計パラメータ。
- **(b) 選択的freeze**: MolmoAct2の公式training codeはLM head・embedding等の
  選択的freezeに対応しており、言語能力の保護に使える。ただしEOS周辺確率方式の
  タイミング学習にはLM本体への勾配が必要なため、freezeの範囲とはトレードオフになる
  （完全freezeが必要ならタイミングヘッドをフォールバックに使う。前節参照）。
- **(c) 外部ナレータVLM（フォールバック）**: SNVLAはタイミング判定のみを行い、実況
  テキストは別のVLMが生成する。MolmoAct2の出発点である `allenai/Molmo2-ER`
  （ロボットfine-tuneの影響を受けていないembodied-reasoning VLM）が同族で好適。
  「単一モデル」という主張は失うが確実に動き、P4の検証段とモデルを共通化できる。
- **(d) 構造化実況**: 実況を自然言語でなくテンプレート/イベント計数にする。言語生成
  能力自体が不要になる。後述のablation P1-E4と同一線上にあり、「記憶に必要なのは言語の
  意味内容か、イベントの回数か」の検証を兼ねる。

fine-tune後の言語能力はP1-E2（生成実況の情報量分析）で常時定量監視する。

#### 第2段階の検証機構 — 自己学習の破綻防止

検証なしの自己学習は、誤実況の自己増幅（ハルシネーションの固定化・語彙の縮退）で
破綻し得る。SNVLAを汎用VLAとして成立させるため、検証の核は**タスク非依存**の
3層で構成し、擬似ラベルはこれを通過したものだけをテキスト教師として採用する:

1. **VLM判定器（主検証）**: P4パイプラインの検証段を流用し、実況前後のフレーム画像・
   タスク指示・実況履歴を与えて判定する。判定は**タイミング妥当性**（この瞬間は
   実況すべき境界か）と**内容正確性**（実況文が観測と整合するか）の2軸で行い、
   タイミング不当のサンプルは全面棄却、内容のみ不正確のサンプルはタイミング教師として
   再利用する（後述）。判定器にはロボットfine-tuneの影響を受けていない
   `allenai/Molmo2-ER` 等を用い、被検証モデルとの重み共有による確証バイアスを避ける。
2. **自己整合性フィルタ**: 同一フレームから温度サンプリングで複数の実況を生成し、
   意味的に一致するもののみ採用する（多数決）。加えて `Progress:` 履歴との矛盾
   （既に報告した事象の重複報告、順序の逆転など）をVLM判定器への含意プロンプトで検出する。
3. **不確実性フィルタ**: 生成時のトークンエントロピー（既にログ済みのメトリクス）が
   閾値超過のサンプルを棄却する。

タスク構造が既知の場合（例: カウントタスクでの数値とイベント通し番号の照合）は、
**決定的バリデータをオプションのプラグイン**として追加できるが、これは特定タスクでの
検証を安価にする加速器であって設計の核ではない。汎用性の主張はタスク非依存の3層のみで
成立させ、P1-E6でプラグインなし構成の性能を必ず測る。

内容正確性のみ不合格のサンプルも**タイミング教師（EOS二値損失）としては採用できる**
ため、データが無駄にならない（タイミング妥当性はVLM判定器が独立に保証している）。
さらにラウンドごとに、(i) held-outの人手実況との一致率、
(ii) 生成実況の語彙多様性、(iii) タスク成功率を監視し、劣化が検出されたら前ラウンドの
チェックポイントへ巻き戻す。

#### リスクと対策

- **シードだけでは書式・接地が不十分**: `Progress:` の既存履歴がfew-shot例として後続
  生成を誘導するが、履歴が空の初回実況が最も脆い。シードには各エピソード冒頭の実況を
  必ず含める。
- **自己学習の分布縮退**: 上記の検証機構+ラウンド監視で対処。縮退が観測された場合は
  シード比率を上げるか、(c) 外部ナレータへ切り替える。
- **言語能力の維持不足**: (a)のco-training比率を上げるか(b)のfreeze範囲を広げる。それでも不足なら(c) 外部ナレータへ。

| P1-E2 | 生成実況の情報量・言語能力分析 | 生成文からの投入回数復元精度、held-out人手実況との一致率、語彙多様性 |
| P1-E3 | シードのタイミング付与をproprioceptionイベント検出で自動化した場合のend-to-end（人手アノテーション完全ゼロ構成） | 成功率 |
| P1-E4 | 履歴の表現形式: 自然言語 vs イベント計数（構造化） | 成功率、カウント誤差 |
| P1-E5 | アノテーションコスト実測 | 収集時間/エピソード |
| P1-E6 | 検証機構のablation（タスク非依存3層それぞれの有無、決定的プラグインの有無） | 自己学習ラウンド経過後の成功率・擬似ラベル精度 |

**採用判断基準**: シードk≤25%でフル実況教師との成功率差が±5pt以内、かつ自己学習
ラウンドを重ねてもP1-E2の各指標が劣化しないこと。

### P2: 実況の非同期並列生成（Asynchronous Narration）

対応課題: C2（主）

#### 動機

実況フレームは0.659s（通常0.415s）と+0.24sの遅延がある。OneTwoVLAは推論中に行動が
停止し、StreamVLAは遷移時のみ推論する。「**行動を止めずに実況し続ける**」設計は
両者と異なる差別化点であり、遅延を実質ゼロ化する。

#### 鍵となる仮説

実況が記憶として効く経路は「次回以降のprefillへの履歴再注入」であり、現行実装が行って
いる「行動生成が直前の実況トークンへKVキャッシュ経由で注意する」即時条件付け
（`modeling_snvla.py:620` 付近）は必須ではない。

#### 学習の変更

現行の実況フレームのターゲット `[BON] 実況 [EOS] [BOA]`（+拡散損失）を2サンプルに分解:

1. **実況サンプル**: ターゲット `[BON] 実況 [EOS]`、テキスト損失のみ
2. **行動サンプル**: `current_narration` を `previous_narrations` 末尾に移した履歴を
   持つprefix + ターゲット `[BOA]` + 拡散損失

これにより「行動は常に完了済みの実況履歴のみに条件付けされる」よう学習する。
既存データセット（`0xNOY/so101_wn_aug`）から機械的に変換でき、再収集は不要。

なお、本節のターゲット表記は現行のBON/BOAアーキテクチャを前提に単独適用可能な形で
記述している。P1と統合する場合はBON/BOAがEOS周辺確率方式に置き換わり、
「分解された2サンプル」の構造（実況はテキストのみ・行動は履歴のみに条件付け）は
そのまま引き継がれる。

**部分実況条件付け拡張**: さらに、`current_narration` をランダム長で打ち切った接頭辞を
prefixに含めた行動サンプルを混合する。これにより行動生成は「実況なし〜部分実況〜完全な
実況履歴」の全域に頑健になり、後述する「生成途中の実況KVの参照」が学習分布の内側に入る。
現行の「全文への即時条件付け」と改訂案の「完了履歴のみへの条件付け」の中間を連続的に
カバーする位置づけである。

#### 推論の変更

1. prefill → モード決定（P1統合時はEOS周辺確率方式）
2. narrate判定でも**即座に** `_act` をprefillのKVキャッシュから実行し、行動キューを充填
3. 実況の自己回帰デコードは別スレッドで継続（チャンク実行中のGPUアイドル時間を利用。
   実況スレッドはprefill KVを読み取り専用で共有し、行動側は独立にBOA分を追加するため
   キャッシュ競合はない）
4. 完了した実況は次サイクルのprefillの `Progress:` に合流

チャンク実行は n_action_steps=15 ≒ 0.5s あり、実況生成（~0.24s）は完全に影に隠れる。
期待効果: 実況フレームの推論サイクル 0.659s → 0.415s。

**生成途中の実況KVの行動への反映**: 完了済み履歴のみへの条件付けを基本としつつ、
生成途中の実況を行動生成が参照する経路を追加する:

1. 次のprefill時点で実況が未完了なら、待たずに**部分文字列**を履歴として使用し、
   残りの実況は打ち切るか次サイクルへ持ち越す
2. RTC併用時は、チャンク実行中の再計画が生成途中の実況トークンのKVを参照する
   （実況スレッドとKVキャッシュを共有）。「生成途中の思考を行動が覗き見る」機構であり、
   同期/完全非同期の二択にない中間的な即時性を与える

いずれも学習側の部分実況条件付け拡張（前節）により学習分布の内側に収まる。
部分文（例: 「2回目を」で切れた文）が誤解を誘発するリスクは、打ち切り位置を
ランダム化した学習と P2-E1 の評価で検証する。

#### 資源とタイミングの分析

本節の数値（推論時間・トークン数・KVサイズ）は現行π0.5実装（PaliGemma 2B）の実測と
構成に基づく。P0移行後もオーダーは同等と見込むが、P2-E2 / P2-E5でMolmoAct2構成として
再計測する。

- **VRAM**: 実況ブランチの増分KVは最大50トークンであり、PaliGemma 2BはMQAのため
  1MB未満で無視できる。ただしHuggingFaceの `DynamicCache` は追記時に `torch.cat` で
  新テンソルを生成するため、実況ブランチと行動ブランチが共有prefix KVからそれぞれ
  延長すると **prefix KV（~1500トークン ≒ 数十MB）が枝ごとに複製される**。
  絶対量は小さいが、実装時に複製の有無を確認し、P2-E5でピークVRAMを実測する。
  同時活性化については、実況デコード（B=1・1トークン）は小さく、行動デノイズと
  重なってもピーク増は限定的と見積もる。
- **実況生成が遅くなる懸念**: 現行実装ではチャンク実行（キュー消化 ≒0.5s）中の
  GPUはアイドルであり、非同期実況はこのアイドル時間に走るため通常は競合相手がいない。
  実況(~0.24s) < キュー消化(0.5s) なので、次のprefill開始前に完了する見込み。
  競合が生じるのはRTC併用時（次チャンク生成と重なる）のみで、対策は
  (i) 実況側CUDAストリームの優先度付け、(ii) `max_narration_length=50` の上限維持、
  (iii) 未完了時は部分文字列で持ち越し（上記の部分実況機構をそのまま使う）。
  P2-E2で「実況完了までのwall-clock時間」を計測し、同期版の生成時間に対して劣化が
  ないことを確認する。

#### リスクと対策

- **実況反映の1チャンク遅れ**: 行動が新実況を参照できるのは次prefillから（最大0.5s遅れ）。
  サブタスク境界直後のチャンクが古い履歴で生成される。影響をP2-E1で定量化し、必要なら
  RTC（`rtc_config` が既にある）+ 生成途中KV参照で早期に再計画する。
- **実装の複雑さ**: torch.compileとスレッド並列の相性に注意。PythonのGIL下でも
  CUDA演算中はGILが解放されるため、実況デコードとチャンク実行の並列は実害が小さい見込み。
  compileが問題になる場合は実況パスのみeagerで動かす。
- **即時条件付けが実は効いていた場合**: 成功率が低下しうる。これ自体が「実況は
  どの経路で効くのか（即時注意 vs 履歴再注入 vs 途中参照）」を切り分ける科学的知見となる。

#### 検証実験

| ID | 内容 | 主要指標 |
|---|---|---|
| P2-E1 | 同期版 vs 非同期版 vs 非同期+途中KV参照 | 成功率、カウント誤差（各30試行） |
| P2-E2 | 推論サイクル時間分布と実況完了レイテンシ | 平均/分散/最大、実況完了までのwall-clock時間 |
| P2-E3 | 学習ターゲット分解のみ（推論は同期のまま）のablation | 学習変更と推論変更の寄与分離 |
| P2-E4 | RTC併用 | チャンク境界の滑らかさ、成功率 |
| P2-E5 | ピークVRAM実測（同期版比較） | `torch.cuda.max_memory_allocated` |

**採用判断基準**: 成功率が同期版との差±5pt以内で、実況フレームの推論サイクルが
非実況フレームと同等になること。

### P3: ハルシネーション対策 — 実況の意味論を「観測記述」に変更

対応課題: C4（主）

#### 動機

論文の失敗分析で、未完了動作への誤完了実況がタスク失敗の主要因と特定済み。
根本原因の仮説: 現行アノテーションでは実況が「動作の意図・進行」と曖昧に結びついて
おり、モデルが視覚確認なしに**手続き的に**完了を宣言することを学習してしまう。

#### 設計 — 3層の防御

1. **データ規約の変更（主対策）**: 実況アノテーションを「結果を視認できるフレーム以降」
   に限定する規約に変更（豆がボウルに入った映像が写ってから `n` を押す）。
   実況の意味論を「意図宣言」から「観測記述」へ移す。既存データは、実況タイミングを
   結果視認フレームへシフトする補正で再利用可能（P4のパイプラインで自動補正）。
2. **不確実性ゲーティング（対症療法）**: 実況開始確率（現行実装では `prob_bon`、
   P1統合後は `1 − P(EOS)`）と生成エントロピーを使い、低確信の実況を保留して
   次サイクルで再判定。閾値は評価データのROCで決定。実装コストはほぼゼロ。
3. **確認行動の学習（発展）**: デモ収集時に実況前の「確認注視」（結果を視認できる
   位置での短い静止。豆すくいならボウル上）を含め、視覚的裏取りの行動パターン自体を
   学習させる。規約はタスク非依存で、静止位置の具体化のみタスクごとに行う。

#### 検証実験

- **ハルシネーション率**を主要指標として新設: 生成実況と実環境状態の一致を
  VLM + 人手サンプリングで判定
- 規約変更前後のモデルで、ハルシネーション率・成功率・カウント誤差を比較
- ゲーティング閾値のsweep（実況見逃しとのトレードオフ）

**採用判断基準**: ハルシネーション率が有意に低下し、成功率・カウント誤差が
非劣化であること。

### P4: VLM自動アノテーションパイプライン

対応課題: C3（P1の基盤）、C4（既存データ補正）

#### 動機

P1が研究の本命だが、そのシードデータ生成・第2段階の擬似ラベル検証（VLM判定器）・
P3の既存データ補正のすべてに使える基盤ツール。単独でも、既存の実況なしデモ資産を
実況付きデータセットに変換できる。

#### 設計 — proprioception × VLM のハイブリッド2段構成

VLM単独のサブタスク分割は精度が低い（F1 ~0.17）ため、proprioceptionで候補を絞る:

1. **候補検出（proprioception）**: グリッパー開閉イベント + 関節速度の極小点から
   サブタスク境界候補を抽出。豆すくいのような構造化タスクでは境界のほぼ全てが
   これらのイベントと一致する見込み。
2. **検証 + 文生成（VLM）**: 候補フレーム前後の画像（top/wristカメラ）+ タスク指示を
   VLMに渡し、(a) 境界としての妥当性検証、(b) 実況文の生成（P3の「観測記述」規約を
   プロンプトで強制）を行う。VLMはP1の判定器と共通の `allenai/Molmo2-ER` を主候補とし、
   精度不足の場合はより大きなVLM（Qwen-VL系ローカル、またはAPI）に切り替える。
3. **出力**: `current_narration` / `previous_narrations` 列を書き込んだLeRobot
   データセット。既存の `snvla-rewrite-dataset-text` / `snvla-augment-narrations` に接続。

#### 検証実験

人手アノテーション済み `so101_wn` をground truthとして:

- タイミング一致率（許容誤差 ±Nフレーム、N=5,15,30）
- 文の意味一致（カウント情報の正誤を自動判定）
- 自動アノテーションのみで学習したモデルの成功率（人手版との差）

**採用判断基準**: 自動アノテーションのみで学習したモデルの成功率が人手版との差
±5pt以内であること。

### P5: シミュレータ対応基盤

対応課題: 全提案の開発円滑化（横断基盤）、C7（評価拡充）

#### 動機

現状、SNVLAの学習・デバッグはすべて実機（SO-101）に依存しており、実況機能の挙動確認
ひとつにもロボットの設営とテレオペが必要である。シミュレータ上で実況機能をデバッグ
できるタスクを整備し、開発の反復を高速化する。

シミュレータには実機にない決定的な利点がある: **スクリプトポリシー（scripted expert）
はサブタスク境界の真値を知っている**。したがって:

- 教師データ収集は人間のテレオペなしで完全自動化できる
- 実況タイミング+テキストは、シム状態のイベント判定（例: 物体がかごに入った）から
  正確に自動付与できる。P3の「観測記述」規約もプログラムで厳密に強制できる
- 真値イベントログをデータセットに保存すれば、実況タイミングのPR曲線評価
  （P1のタイミング学習の直接評価）ができる
- 生成実況とシム状態の真値を照合すれば、**ハルシネーション率を人手やVLMなしで
  自動計測**できる（P3の主要指標の自動化）

#### タスク設計 — 記憶依存タスクの要件

実況機能のデバッグには、Perceptual Aliasingを含み実況（記憶）がないと解けないタスクが
必要である。候補:

- **カウント系（主案）**: 「ブロックをN個かごに入れる」。投入済み個数は視覚的に
  判別しにくく（遮蔽あり）、実機の豆すくいタスクと同型の記憶要求を持つ
- **順序系**: 「2種類のオブジェクトを交互に運ぶ」「決められた順序でボタンを押す」。
  カウントとは別類型のPerceptual Aliasingで、C7の評価拡充にも使える

スクリプトエキスパートはwaypointベースのpick-and-placeで構成し、イベント完了の
シム判定直後のフレームに実況を付与する。

#### シミュレータの選定

- **主案: LIBERO（robosuite/MuJoCo系）+ カスタムBDDLタスク**。理由:
  (i) `allenai/MolmoAct2-LIBERO-LeRobot` チェックポイントが存在し、P0のpolicy移植を
  「既知の動作点」に対してデバッグできる、(ii) VLA標準ベンチマークであり
  C7（査読耐性のある評価）に直結する、(iii) BDDLでカスタムタスクを定義できる。
  LeRobot 0.6とのenv統合状況は着手時に確認する。
- 代替1: **ManiSkill3**。GPU並列で高速、motion-planningによるデモ自動生成の仕組みが
  公式にある。大規模スケール実験（P1の自己学習ラウンド等）が必要になった段階で検討。
- 代替2: **MuJoCo + SO-100/101モデル**（mujoco_menagerie）。embodimentがP0主案
  （`MolmoAct2-SO100_101`）と一致するデジタルツイン。sim2realの一貫性が必要になった
  段階で検討。環境・タスク・エキスパートをすべて自作するコストが最も高い。

#### 各提案への寄与

| 提案 | シミュレータが提供するもの |
|---|---|
| P0 | MolmoAct2-LIBERO-LeRobotを基準にしたpolicy移植のデバッグ |
| P1 | 真値タイミングでのPR評価、自己学習ラウンドの高速反復、P1-E6の検証機構較正 |
| P2 | 非同期タイミング挙動の再現的デバッグ、レイテンシ計測 |
| P3 | ハルシネーション率の自動計測（シム真値と照合） |
| P4 | 真値付きデータによるアノテーション精度評価 |

さらに「自動収集 → 学習 → 評価」のスモークテストを1 GPUで回る回帰テストとして整備し、
以後の全変更の検証に使う。

#### 検証実験（整備完了の判断基準を兼ねる）

| ID | 内容 | 主要指標 |
|---|---|---|
| P5-E1 | スクリプトエキスパートによる自動データ収集（テレオペなし） | 収集スループット（エピソード/時）、実況付与の正確性（真値一致率） |
| P5-E2 | 自動収集データで学習したSNVLAのシム内評価 | 成功率（実況ありが実況なしablationを大きく上回ること = 記憶依存タスクとして成立） |
| P5-E3 | スモーク回帰テスト（収集→学習→評価の縮小版） | 1 GPU・一定時間内での完走 |

**採用判断基準**: P5-E2で「実況あり ≫ 実況なし」の差が再現されること
（実機実験と同じ構図がシムで成立して初めてデバッグ基盤として意味を持つ）。

## 5. 概要レベル提案（優先度: 中〜低）

- **C5 コンテキスト圧縮**: 実況履歴の古い項目の要約・削除、またはprefill KVキャッシュの
  差分更新（履歴は追記のみなので、画像以外のKVは再利用可能性がある。ただし画像トークンが
  毎フレーム変わるため位置がずれる点に注意）。現行タスク（〜10実況）では顕在化して
  いないため、長時間タスクへ拡張する際に着手する。
- **C6 exposure bias**: P1の第2段階（自己教師拡張）で実質的に対処される。P1を採用しない
  場合は、教師履歴へのノイズ注入（実況の欠落・言い換え）による頑健化が安価な代替。
- **C7 評価拡充**: P5のシミュレータ基盤（LIBERO + カスタム記憶タスク）を評価
  ベンチマークとしても整備し、再現性ある比較を可能にする。実機では豆すくいに加え
  「元の位置に戻す片付けタスク」等、Perceptual Aliasingの別類型を追加。
  P1/P2の主張を単一タスクで行うと査読耐性が弱い。

## 6. ロードマップ

```
P5（シミュレータ対応基盤）──実況デバッグ/自動データ収集/回帰テスト──→ 全提案の開発を高速化
P0（MolmoAct2移行）──全提案の土台（言語生成能力・SO-101事前学習）──→ P1〜P4
P4（自動アノテーション基盤）──シード生成/擬似ラベル検証/データ補正──→ P1（自己教師アノテーション拡張）
P3（観測記述規約）──規約は以後のすべてのデータ収集に適用──→ P1・P4
P1 + P2 ──統合──→ 次世代SNVLA（EOS周辺確率のモード決定 + 非同期実況 + 途中KV参照）
```

- **最初の一手（開発基盤）**: P5のシミュレータ対応を最優先で整備する。LIBERO +
  カスタム記憶タスク + スクリプトエキスパートによる自動データ収集（テレオペ不要）で、
  以後の全提案のデバッグと検証を実機なしで高速に回せるようにする。P0のpolicy移植も
  `MolmoAct2-LIBERO-LeRobot` を基準にシム上でデバッグする。
- **短期（土台と性能・実用の改善）**: P5の上でP0のMolmoAct2移行を行い、続いて
  P3の規約変更 + P2の非同期化により成功率（ハルシネーション削減）と遅延（+0.24s解消）を
  改善する。P2-E3（学習ターゲット分解のみ）は既存π0.5資産のデータ変換だけで先行検証
  でき、P5・P0と並行して着手できる。
- **中期（新規性・スケール）**: P4で基盤整備 → P1で「自己教師による実況アノテーション
  拡張」を主張する次論文。P1の自己学習ラウンドはまずP5のシム上で較正してから実機
  データに適用する。P2の「行動を止めない実況」を統合すれば、OneTwoVLA/StreamVLAに
  対してデータ効率と実行効率の両面で差別化した統合アーキテクチャとして提示できる。
