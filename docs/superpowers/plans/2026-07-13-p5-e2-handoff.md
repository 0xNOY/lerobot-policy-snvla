## 2. DGX 2GPU構成でのOOM（解決済み）

ユーザー指示: **以降はGPU 2枚のみ（`CUDA_VISIBLE_DEVICES=2,3`）**。

phase-1（4GPU・SHARD_GRAD_OP・batch16×4）はstep 2843まで健全に走り
`outputs/train/snvla_t1_n3_v3_aug/checkpoints/002500` が生成済み（DGX上）。
FSDPのresumeは壊れている（optimizer復元で device/dtype エラー）ため、
**ckpt2500のpretrained_modelを初期値にした新規ラン（fresh optimizer）**で続行する方針。

2GPUでの試行はすべて **OOM（40GB/GPU）**:

| 試行 | 構成 | 結果 |
|---|---|---|
| p2-a | SHARD_GRAD_OP, batch16×2 | OOM（forward中の `txt_logits.to(float32)` で3.54GiB確保失敗） |
| p2-b | FULL_SHARD, batch16×2 | 同上（fsdp_sharding_strategyが効いていない可能性あり） |
| p2-c | SHARD_GRAD_OP, batch8×2 | OOM（静的38.5GiB確保後、18MiBで失敗） |
| p2-d | FULL_SHARD + reshard_after_forward, batch8×2 | 同上 |

観察: accelerateが「Upcasted low precision parameters ... because mixed precision」と警告
→ **全パラメータがfp32化**されている（4.14B×4B=16.6GB）。b/dでもピークが同じことから
shardingが実際に効いていない疑いが強い（accelerate CLIフラグとFSDP2の対応を要確認。
`--fsdp_version` フラグが存在する。lerobot-train側のAcceleratorがプラグイン設定を
上書きしている可能性も要調査: `.venv/lib/python3.13/site-packages/lerobot/scripts/lerobot_train.py`）。

**有望な対策（順に試す価値）**:
1. **lm_headの全系列適用をやめる**: `src/lerobot_policy_snvla/modeling_snvla.py` の
   `SNVLACore.forward` は `lm_head(prefix_out)`（画像~1024+テキスト~160トークン全部、
   vocab 257k）→スライス→fp32化、の順。**言語部分にスライスしてからlm_head**にすれば
   logitsメモリが約1/8になる（3.5GiB→0.5GiB、正当性はloss計算がlanguage_seq_lenしか
   使わないため自明）。fp32化もCE直前のみでよい
2. accelerateのFSDP設定を`--fsdp_version 2`や設定ファイル（`accelerate config`のyaml）
   経由で正しく効かせ、FULL_SHARDの実効を`mem_gb`ログで確認する
3. `--fsdp_offload_params true`（optimizerをCPUへ。遅くなるが確実）
4. `--policy.freeze_vision_encoder=true`（対象パラメータ減。SNVLAConfigはPI05Config継承
   なのでフラグは存在する。言語・action expertは学習可能なまま）

### 解決内容とベンチマーク

本命の対策1に加え、根本原因だったFSDPのbf16 parameter upcastを回避した。root FSDPの
native bf16でOOMを解消した後、decoderを通常の`forward`で呼べる`JointDecoderLayer`に
分離してlayer FSDPを有効化した。CPU offloadは20.5 GiBまで下がったが39.1 s/step、
Linear単位FSDPは18.33 GiBだが約7.3 s/stepで、いずれも不採用。

| 構成 | 定常step時間 | PyTorch peak | 判断 |
|---|---:|---:|---|
| root FSDP eager（初期） | 約2.69 s | 約34.45 GiB | OOM解消の基準 |
| layer FSDP eager | 約1.76 s | 23.55 GiB | 安定 |
| SDPA + loss-token限定、ckpt interval=1 | 1.22 s | 23.55 GiB | 安全側 |
| 同 interval=2 | 1.10〜1.11 s | 29.73 GiB | **採用** |
| 同 interval=3 | 1.43〜1.54 s | 33.74 GiB | 遅く余裕も小さいため不採用 |

RTX 3090では単GPUのcompile/SDPA/fused QKV単体デバッグが可能だった。一方、AdamWを含む
本学習は24GBではOOMするため、FSDP統合確認は許可されたDGX GPU 0,1で行い、本番だけを
GPU 2,3へ移した。SDPAはlocal equivalenceでloss差 `8.96e-05`。FlashAttention強制は
dense non-null attention maskをサポートせず不採用とし、PyTorch SDPA autoを用いる。

## 3. 現行の本番学習コマンド（DGX）

```bash
ssh dgx
cd ~/Workspaces/lerobot-policy-snvla
# 注意: .venv/bin/* のshebang自体はDGX上では正常（ローカルと違い破損していない）
# ただし必ず CUDA_VISIBLE_DEVICES=2,3 を付ける（DGX Display GPUと0,1は使用禁止）
CUDA_VISIBLE_DEVICES=2,3 PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True HF_HUB_OFFLINE=1 \
TORCHINDUCTOR_CACHE_DIR=$HOME/.cache/torchinductor_snvla_production_advanced \
.venv/bin/accelerate launch \
  --num_processes=2 --use_fsdp --fsdp_version=1 \
  --fsdp_sharding_strategy=FULL_SHARD \
  --fsdp_auto_wrap_policy=TRANSFORMER_BASED_WRAP \
  --fsdp_transformer_layer_cls_to_wrap=JointDecoderLayer \
  --fsdp_backward_prefetch=BACKWARD_PRE \
  --fsdp_state_dict_type=SHARDED_STATE_DICT \
  --fsdp_use_orig_params=true \
  --mixed_precision=no \
  --module lerobot_policy_snvla.scripts.train_bf16_fsdp \
  --dataset.repo_id=local/t1_n3_v3_aug \
  --dataset.root=$HOME/datasets/t1_n3_v3_aug \
  --policy.type=snvla \
  --policy.pretrained_path=$HOME/Workspaces/lerobot-policy-snvla/outputs/train/snvla_t1_n3_v3_aug/checkpoints/002500/pretrained_model \
  --policy.push_to_hub=false \
  --policy.compile_model=true \
  --policy.compile_cudagraphs=true \
  --policy.training_padding_length=256 \
  --policy.max_text_loss_tokens=16 \
  --policy.attention_backend=sdpa \
  --policy.fuse_qkv=true \
  --policy.gradient_checkpointing=true \
  --policy.gradient_checkpointing_interval=2 \
  --policy.dtype=bfloat16 \
  --policy.optimizer_lr=10.0e-5 \
  --policy.device=cuda \
  --policy.max_state_dim=32 \
  --policy.max_action_dim=32 \
  --steps=15000 --batch_size=8 \
  --log_freq=25 --save_freq=2500 \
  --num_workers=8 \
  --output_dir=outputs/train/snvla_t1_n3_v3_aug_p2 \
  --wandb.enable=false
```

- lr=10e-5/batch8×2 はglobal batch 16でのスケール済み値（phase-1は20e-5/batch64）。

- `--output_dir` が既存だとエラー → `rm -rf outputs/train/snvla_t1_n3_v3_aug_p2` してから
- 健全性の確認: ログに `All keys loaded successfully!`（**必須**。

## 6. 環境の落とし穴（全既知事項）

- ローカル3090: 学習不可（optimizer初期化でOOM）。評価・データ処理のみ
- ローカル `.venv/bin/*` shebang破損 → `.venv/bin/python -m` 形式
- DGX: `CUDA_VISIBLE_DEVICES=2,3` のみ使用可。torchは `--torch-backend=cu128`
  （cu130/cu132はドライバ570でCUDA初期化不可→CPU実行になる）。venvはPython 3.13
- max_state_dim/max_action_dim は **32/32**（7に変えると全重みロード失敗、
  警告のみで学習が進む罠）。実7次元への切り出しは推論側が行う
- compile対象をTransformer全体へ広げるとFSDP collectives/checkpointingとの相互作用で
  再compile・速度低下・Dynamo crashが起きる。現行は固定形状loss reductionのみを
  compile/CUDA Graph化する設計なので、`compile_model=true` / `compile_cudagraphs=true`必須
- LIBERO: `MUJOCO_GL=egl`、初回importの対話は `echo N |` で回避（解決済み環境）
- 評価seedは 10_000_000〜（収集seed帯 worker*100_000 と非重複）
- `snvla_pi05base_init`（~/models/、DGXにも同パス）はpi05_base重み+SNVLA processor JSON
  の初期化チェックポイント。processor JSONは32/32で再生成済み
- 増強データでは同一実況文が「実況中レイアウト」（current_narration、[BON]〜[BOA]）と
  「履歴レイアウト」（Progress:内）の両方で学習される。デバッグ時の条件付け比較は
  フレーム0（実況center）等、レイアウトが学習分布と一致する点で行うこと
