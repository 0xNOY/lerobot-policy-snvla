## 1. データセット: 実況フレーム増強（forward-only）

`snvla-augment-narrations` に `--forward-only` モードを追加し、
`local/t1_n3_v3` → `local/t1_n3_v3_aug` を生成した（window=10）。

- 伝搬は `[center, center+10]` に限定（centerより**前**に伝搬しない）。
  ` (done)\n` 断片のcenterはシム真値イベントのフレームであり、前方伝搬を許すと
  「結果視認前の完了実況」を学習してC4ハルシネーションの原因になる（P3規約の保護）
- 検証結果: 全50エピソードでストリーム一致、各断片の初出フレームが元のcenterと
  完全一致（後方伝搬ゼロ）、`previous_narrations` のrun内不整合ゼロ
- 実況フレーム率: 1.68% → **17.21%**

## 2. 学習構成（確定値）

ローカルRTX 3090 24GBはAdamW momentsのoptimizer初期化時にOOMで**学習不可能**
（batch=1でも静的コスト~21.6GBが支配的）。DGX Station A100（4×A100 40GB）へ移行した。

| 項目 | 値 |
|---|---|
| ハード | DGX Station A100（`CUDA_VISIBLE_DEVICES=0,1,2,3` 必須） |
| 並列化 | accelerate FSDP SHARD_GRAD_OP、bf16 mixed precision |
| batch | 16/GPU × 4 = 64（paper構成と同じglobal batch） |
| steps | 10000（≒16.6 epoch）、save_freq=2500 |
| lr | 20e-5（paper値、cosine decayはstepsに自動スケール） |
| 初期化 | `~/models/snvla_pi05base_init`（pi05_baseのコピーにSNVLA processor JSONを差し替え） |
| 次元 | max_state_dim=32 / max_action_dim=32（pi05_baseネイティブ。7に縮めると全重みロード失敗） |
| その他 | compile無効（dynamoクラッシュ）、gradient checkpointing有効、4.06〜4.6 s/step |

学習は健全: loss 0.124→0.07台（~1.5 epoch時点）、checkpoint 2500のteacher forcingで
実況を完全予測、action回帰MSE 0.004〜0.02（正規化空間）。

**phase-1はstep 2843で停止**（FSDP resumeがoptimizer復元のdevice/dtypeエラーで壊れて
おり、以後はckpt2500のweights+fresh optimizerの新規ランで続行する方針）。ユーザー指示で
以降はGPU 2枚（`CUDA_VISIBLE_DEVICES=2,3`）に制限された。当初の2GPU構成
（SHARD_GRAD_OP/FULL_SHARD × batch16/8）は40GB OOMだったが、次節の対策で解消した。
試行履歴と原因分析は引き継ぎ文書 `2026-07-13-p5-e2-handoff.md` §2を参照。

### 2.1 phase-2最終構成（2026-07-13）

OOM原因を、(1)画像を含む全系列への巨大`lm_head`適用、(2)Accelerate FSDPのbf16 parameter
upcast、(3)root-only FSDPによるsharding粒度不足に分解して解消した。`lm_head`は言語部分、
最終的にはloss対象末尾16 tokenだけに適用し、fp32化をCE直前に限定。decoderを
`JointDecoderLayer`化してFULL_SHARD可能にし、旧checkpointキーも互換ロードする。

| 項目 | phase-2確定値 |
|---|---|
| GPU | `CUDA_VISIBLE_DEVICES=2,3`、A100 40GB ×2 |
| 並列化 | FSDP v1 FULL_SHARD、`JointDecoderLayer` wrap、bf16 parameter/autocast |
| batch / lr | 8/GPU ×2 = 16、10e-5 |
| 初期化 | phase-1 checkpoint 002500 weights、fresh optimizer |
| 高速化 | 固定padding 256、SDPA、fused QKV、checkpoint interval=2 |
| compile | 固定形状loss reductionのみcompile + CUDA Graph |
| 次元 | max_state_dim=32 / max_action_dim=32 |
| 実測 | 1.10〜1.12 s/step、14 samples/s、PyTorch peak 29.73 GiB |

全Transformer graphのcompileはFSDP collectivesとactivation checkpointingにより遅く、
DDP+compileはoptimizer初期化でOOMした。CPU offloadも39.1 s/stepのため不採用。採用構成は
初期root FSDP eager（約2.69 s/step）より約58%、最適化前phase-2（約1.78 s/step）より
約37%高速。非simテストは `36 passed, 12 deselected`。

当時の本番runの最終監査記録はstep 50: loss 0.068、1.122 s/step、14 samples/s、29.73 GiB。
両rankで `All keys loaded successfully!` を確認し、ランダム初期化警告、OOM、runtime error、
compile recompile-limitはいずれも出ていない。最適化前runはstep 975で停止し、ログを
`outputs/train/snvla_t1_n3_v3_aug_p2.pre_advanced_opt_975.log` に退避した。

## 3. シム評価が検出した推論バグ（P5基盤の成果）

checkpoint 2500の先行評価で、**実機なしで**以下の移植バグ2件を検出・修正した。
いずれもpaper時代のfork（旧transformers）では正しく、transformers 5.xへの
移行で顕在化したもの。実機評価でも同様に発生していたはずである。

### 3.1 denoise_stepのKVキャッシュ増殖（クラッシュ）

transformers 5.xは `use_cache=False` でもCacheオブジェクトにキーを追記する。
SNVLAの `denoise_step` オーバーライドには本家pi05にある
`clone_past_key_values()` が欠けており、denoiseのたびにキャッシュが
chunk_size分成長し、attention mask長不一致でクラッシュした。→ 本家と同じcloneを追加。

### 3.2 埋め込みの二重スケーリング（実況・行動の崩壊）

transformers 5.xのGemmaは埋め込み層自体が `sqrt(hidden)` スケールを内蔵する
（`GemmaScaledWordEmbedding`）。`_narrate_step`/`_act` の手動スケーリングが
**~45倍の二重スケール**となり、実況デコードが語彙外トークンの繰り返し
（p=1.0で「 increa」）に、BOA経由の行動条件付けも崩壊していた。
teacher forcingは `embed_prefix` 経由のため無傷で、学習lossだけ見ても
検出できない罠だった。→ スケール内蔵の `embed_language_tokens` に統一。

### 3.3 教訓: 条件付けレイアウトの取り違え（誤診の記録）

デバッグ中、「インクリメンタルに構築したKVキャッシュが行動を壊す」と一時誤診した。
原因は比較実験の欠陥: 増強データでは同じ実況文が「実況中レイアウト
（current_narration、[BON]〜[BOA]ターゲット）」と「履歴レイアウト（Progress内）」の
両方で出現し、フレームによってどちらが学習分布かが異なる。フレーム100
（履歴レイアウトが正）に実況中レイアウトで条件付けて「壊れた」と判定していた。
公平な比較（フレーム0、実況center）ではインクリメンタル経路 err 0.163 /
一括再prefill経路 err 0.170 で同等に良好であり、経路のバグは存在しない。

## 4. checkpoint 2500（4 epoch相当）の先行評価

修正後の評価（10エピソード、実況あり、seed 10,000,000〜）:

| 指標 | 値 |
|---|---|
| 成功率 | 0/10 |
| 平均placed | 0.0 |
| 実況の形式 | **完璧**（so101_wn互換の断片ストリームを正しく生成） |

観察された失敗モード: 行動精度が把持に足りず、ロールアウト中の分布ドリフトで
モード決定が崩れ、行動の進捗なしに `(done)` が積み上がる**手続き的ハルシネーション
連鎖**（スペックC4と同じ構図）。分布内フレームでの診断では、モード決定は概ね較正
されており（実況centerでBON 1.0、非実況フレームでBOA 0.96+、一部0.24〜0.44と甘い）、
行動回帰も健全なため、当時は**主因はundertraining（4 epoch）**と判断した。
旧方針の「10000 stepまで継続し、checkpoint 5000/7500/10000で再評価」は
**canceled/replaced**であり、実行しない。

なお「行動なしのハルシネーション連鎖」をシム真値で自動観測できたこと自体が、
P5基盤の目的（C4計測の自動化、実況デバッグの高速反復）の実証である。
